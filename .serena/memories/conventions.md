# Conventions

- Documentation is primarily Japanese with English technical/product terms; use ATX heading hierarchy.
- Product-specific documentation uses lowercase kebab-case filenames; reviews use the `review-pawlens-` prefix. A direct-review standalone prototype may use a descriptive filename with spaces.
- Keep requirements, research, UI design, review, and presentation artifacts separate under `docs/`.
- Proposed architecture must be labeled planned until backed by executable source/configuration.
- Safety contract: output uncertainty, limits, observable cues, and safe actions; do not present a diagnosis or definitive translation.
- For product source-of-truth and directory patterns, read `mem:core`.
