## 結論

**現時点では優勝圏とは判定できません。** アイデア、問題設定、UI設計には明確な強みがありますが、審査の入口となる「GPT-5.6 と Codex を使った、審査員が再現できる動作プロダクト」の証拠がまだありません。そのため、現状は Stage 1（viability）を通過できる状態ではありません。

ただし、PawLens の核である **「犬語の翻訳」ではなく、不確実性を表示しながら飼い主の観察と次の行動を支援する解釈アプリ** は差別化されています。来客時の吠え声という一つの場面に絞り、実際の添付入力、推論、確認質問、記録、次回比較までを一気通貫で動かせれば、優勝候補として戦える余地があります。

## 勝ち筋

### 一つの「再来訪」を見せる

PawLensの最も強いデモは、単発の感情推定ではありません。次の流れです。

1. 来客のチャイム直後に吠えた場面を、短い音声と状況入力で記録する。
2. PawLensが断定せず、仮説・確信度・観察点・安全な次行動を4カードで返す。
3. 飼い主が「耳が後ろ」「距離を取った」など実際に確認できたサインを保存する。
4. 次の来客時に再記録し、「以前より回復が早い／距離の取り方が変化した」と、履歴を根拠に比較する。

この一連が動けば、単なるマルチモーダル分析ではなく、飼い主と犬の相互理解を積み上げるアプリだと伝わります。審査上は、技術・デザイン・インパクト・アイデアを同時に見せられる構成です。

### MVPで捨てるもの

提出までに実装対象を増やさないでください。最初の完成形は以下で十分です。

- 場面は **来客時の吠え** のみ。
- 入力は音声、状況、任意の写真のみに限定する。短尺動画、複数の場面、プロフィール管理の作り込みは後回しにする。
- 出力は4カードと確認質問を固定する。
- 保存するのは、確認済み観察サインとタイムスタンプを含む1件の記録だけにする。
- 履歴比較は「前回との違い」を一つ表示できればよい。

## 最優先リスクと改善案

| 優先度 | リスク | 審査への影響 | 具体的な改善 |
|---|---|---|---|
| P0 | **音声だけから感情を断定して見える** | 科学的な信頼性と安全性を失い、「犬語翻訳」の焼き直しに見える。 | 音声を決定根拠にせず、状況・視覚的サイン・飼い主の確認回答と併用する。出力には根拠、代替仮説、観察できない限界を必須フィールドとして持たせる。 |
| P0 | **設計範囲が広く、動く薄片が完成しない** | Stage 1を通過できない。 | 上記の来客シナリオ以外を明示的にバックログへ移す。まず入力→MCP→4カード→確認→保存→比較を完成させる。 |
| P0 | **Apps SDK/MCPとしての実証がない** | 高いデザイン資料でも技術実装の点にならない。 | 公開到達可能なMCPサーバーを用意し、ChatGPTの開発者モードで実接続する。添付入力は `openai/fileParams` を使う設計にし、ツール呼び出し中だけ有効なファイルURLを保存しない。 [MCPサーバーの公式ガイド](https://developers.openai.com/apps-sdk/build/mcp-server) |
| P0 | **提出物・運用証拠が後回し** | 動いても失格・評価不能になり得る。 | 実装開始時点からREADME、テストアカウント、3分動画の台本、ライセンス、英語サマリー、Codex Session IDを提出チェックリスト化する。 |
| P1 | **音声・画像・犬の行動ログのプライバシーが未定義** | 審査員・利用者に不信感を与える。 | 保存対象、保存期間、削除方法、第三者提供の有無を明示する。デモでは実在の住所・人物音声・顔が分かる映像を使わない。 |
| P1 | **医療・行動診断のような助言に見える** | 安全面の懸念が強く、誤誘導のリスクがある。 | 診断・治療・保証を避け、危険サインでは「距離を確保し、必要なら獣医師・行動の専門家へ相談」とする。安全上の固定ルールをモデル出力の前後で検証する。 |
| P1 | **調査数値が強い主張だけで使われる** | インパクトの信頼性を損なう。 | スライドの統計には一次ソース、対象集団、調査年、定義を添える。`research-pawlens.md`で未検証とされる数値は、検証が終わるまで外部向けの主要訴求に使わない。 |
| P2 | **設計とデモの不一致** | デザインの説得力が落ちる。 | 仕様の象徴的な「霧が晴れる」表現は、プロトタイプでは未有効化である。実装するか、発表資料から外して期待値を揃える。 |

## 実装レビュー: 最小アーキテクチャ

要件の4ツール構成は妥当ですが、提出版ではさらに単純化できます。

```text
ChatGPT
  └─ PawLens widget
       ├─ 音声・写真・状況を入力
       └─ MCP tool: analyze_dog_signal
              ├─ 添付をその場で取得・前処理
              ├─ GPT-5.6で構造化解釈を生成
              └─ { hypothesis, confidence, limits, observation_points, next_action, follow_up_question }
       └─ MCP tool: save_observation
              └─ 飼い主が確認した観察サインだけを記録
       └─ MCP tool: get_dog_history
              └─ 前回との比較を返す
```

実装上の判断基準は次です。

- モデルに返すデータと、UIだけに返すデータを分ける。ツール結果には必要最小限の構造化データを返し、録音・画像そのものや不要な個人情報を会話へ返さない。
- ウィジェットは `window.openai` の可否を機能検出し、添付選択・PiP・全画面が利用できない環境でも主タスクを完遂できるようにする。[ChatGPT UIの公式ガイド](https://developers.openai.com/apps-sdk/build/chatgpt-ui)
- 外部接続先は最初から最小化し、CSPの `connect_domains` と `resource_domains` を正確に宣言する。公開提出するアプリでは、実際に到達できるエンドポイントとCSP不整合がよく失格・不承認要因になる。[Apps SDKの提出ガイド](https://developers.openai.com/apps-sdk/deploy/submission)
- 「GPT-5.6が音声を直接どこまで解釈できるか」を憶測で設計固定しない。最初に少数の代表入力で技術スパイクを行い、必要なら音響特徴量＋画像＋状況の解釈へ切り替える。この判断と結果をデモ・READMEに残す。

## 検証計画

### 実装前の技術スパイク

来客時の代表的な4入力について、以下を同じフォーマットで記録します。

| 確認項目 | 合格条件 |
|---|---|
| 添付入力 | 音声・任意写真がツールへ渡り、失敗時にも再試行または代替入力を案内できる。 |
| 出力JSON | 仮説、確信度、限界、観察点、次行動、確認質問が常に存在する。 |
| 非断定 | 犬の一人称発話、感情の確定、医療・治療の指示が出ない。 |
| 根拠の整合 | 音声・状況・確認済みの観察サインにない事実を、根拠として作らない。 |
| 低信頼時 | 「追加観察を提案する」状態になり、強い行動推奨をしない。 |

### デモ前の受け入れテスト

既存の8ケースを使い、各ケースで「表示された」だけでなく次を確認します。

- 構造化フィールドと安全ルールは **8/8** で満たす。
- 分析→確認→保存→履歴比較の通しシナリオは **少なくとも2回**、別の入力で成功する。
- 3人以上が出力を見て、根拠と限界を区別できるかを確認する。
- 5人程度の飼い主テストでは、少なくとも4人が「次に観察するサインを2つ」と「安全な次行動」を言い直せることを目標にする。

最後の二項目は、現時点で達成済みの成果ではなく、収集すべき検証目標です。

## 提出までの優先順位

1. **技術スパイクを完了し、入力形式と推論の責任範囲を決める。**
2. **来客シナリオのMCP接続を完成する。** 分析、確認、保存、比較の薄い縦切りを先に動かす。
3. **安全ガードと失敗状態を実装・テストする。** 不足情報、ファイル失敗、低信頼、危険サインを必ず通す。
4. **実際のChatGPT内でUIを整える。** プロトタイプの良さを移植し、主タスクに不要な演出は削る。
5. **提出パッケージを作る。** 3分動画、英語README、試用手順、ライセンス、プライバシー説明、Codexの作業証跡、テスト結果を同時に揃える。

公式締切は **2026-07-21 17:00 PDT** です。機能を増やすより、締切前に第三者が迷わず再現できる一つの完成体を固定することが優先です。[公式ルール](https://openai.devpost.com/rules)

## 最終判定

**判定: 条件付きで有望、現時点では提出・優勝判定ともに未到達。**

PawLensは、誇張した「翻訳」を避け、飼い主が犬をよりよく観察するための道具へ転換している点が強いです。勝敗を分けるのは、資料の完成度ではなく、次の三点を実演できるかです。

1. 実際の入力を使った、再現可能なGPT App。
2. 不確実性と安全性を、機能として守る構造化出力・ガード・テスト。
3. 一回の推定ではなく、飼い主の確認と履歴が次の理解を良くする体験。

この三点に集中すれば、PawLensは「かわいいペットAI」ではなく、Apps for Your Lifeで評価される、慎重で実用的な生活支援アプリとして差別化できます。
