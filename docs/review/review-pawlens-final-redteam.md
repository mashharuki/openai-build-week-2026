## 2. 新規の重大指摘（過去レビュー未指摘・深刻度順）

### Critical

**C-1. R-1（音声理解）のフォールバック計画は循環依存で、実は機能しない**
- 場所: requirements.md §12 R-1「不足なら、**クライアント側**で基本的な音響特徴（長さ・ピッチ帯・繰り返し）を抽出して補助入力にする」
- なぜ危険か: ChatGPT App において「クライアント」は OpenAI が制御する ChatGPT 本体であり、あなたのコードは走らない。音響特徴を抽出できる場所は自前の MCP サーバだけだが、サーバに生音声が届くかどうかこそが R-2 の未検証事項。つまり **R-1 の Plan B は R-2 が成立する場合にしか存在せず、R-2 が崩れると Plan A と Plan B が同時に崩壊する**。しかも全成果物のワンライナー（"a dog's voice"）・デモ台本・スライド4/5 が音声入力を主役に据えており、崩れた場合の書き換え範囲が最大級。
- どう直すか: **今日、実装より先に** Developer Mode + 任意の既存MCPサーバで「会話添付の音声ファイルがツール引数（`openai/fileParams` 等）として届くか」を30分スパイクで白黒つける。届かないと判明したら即座に入力契約を「飼い主による鳴き声の記述（ガイド付き質問）＋写真＋状況」に差し替え、ワンライナーから "voice" を外す判断を今日中に行う。「音声が主役かどうか」を Day3 に持ち越すことが最大の敗因候補。

**C-2. 差別化の核「2回目の履歴比較」を支える個体同定・永続化の設計が存在しない**
- 場所: requirements.md §6.3「個体は匿名IDで管理」/ §6A.2「会話をまたぐ状態は KV/D1 に個体IDで永続化」
- なぜ危険か: MVP に認証（OAuth）は無い。では KV のキーになる「匿名ID」は**誰がいつ発行し、次の会話でどうやって同じIDに辿り着くのか**が全ドキュメントのどこにも書かれていない。ChatGPT がツール引数として前回のIDを覚えて渡してくれる保証は無く、会話を跨いだ瞬間に「その子の手帳」は空になる。プロダクト約束（PRD §0「一度の推定で終わらず…次回の理解へつなぐ」）とスライド8の物語全体が、未設計の一点に載っている。
- どう直すか: MVPの履歴保証を**「同一会話内」に明示的に格下げ**する（analyze→確認→保存→同会話で2回目、で差分は成立する）。`manage_dog_profile` が返す `dog_id` を `structuredContent` と `widgetState` の両方に埋め、次回ツール呼び出しの引数に含めるようツール記述子で誘導する。judge test card には「同じ会話内で2回試す」手順を明記し、会話跨ぎは「動けばボーナス」扱いにする。

**C-3. 審査員向けの英語“出力”要件がプロダクト要件に存在しない**
- 場所: requirements.md §3.2「MVPは日本語中心＋英語の応募資料」/ FR・ACの例文はすべて日本語 / design-spec §8 は `window.openai.locale` を日付書式にしか使っていない
- なぜ危険か: 過去レビューが指摘した「応募資料の英語化」とは別問題。judge-testable とは**審査員が英語で話しかけ、英語のカードを受け取る**ことを意味する。プロトタイプには en ロケールのシナリオデータが既に存在する（デコードで確認: "Mugi / Shiba Inu / Alertness / Medium"）のに、PRD側に「出力言語はユーザーのロケールに追従する」というFRが無いため、実装時に日本語ハードコードされる危険が高い。審査員が日本語カードを受け取った瞬間、Design と Technical の両軸が同時に減点される。
- どう直すか: FR-13 として「カードの全テキスト（仮説ラベル・限界文・観察ポイント・CTA・追加質問）はリクエスト言語に追従する。少なくとも en/ja」を追加。GPT-5.6 への出力スキーマに `locale` を渡すだけで実現でき、コストは小さい。AC にも英語1ケースを足す。

**C-4. ウィジェット記述子の技術前提が ChatGPT の実仕様と食い違っている可能性（ドラフト注記が下流に伝播している）**
- 場所: requirements.md §6.2「`_meta.ui.resourceUri`」§6.3「MIME `text/html;profile=mcp-app`」「MCP Apps bridge（`ui/notifications/tool-result`）」
- なぜ危険か: これらは汎用 MCP Apps（SEP系）の規約であり、OpenAI Apps SDK が要求してきた規約（`ui://` リソース + `openai/outputTemplate` メタデータ + skybridge 系 MIME）とは体系が異なる。PRD 自身は「本節はドラフト、$openai-docs で再確認」と正しく注記しているが、§6.2 の表・6A.7 の流用計画・design-spec 6章は**断定調**で書かれており、実装者（Codex）がドラフトのまま実装して「ツールは動くがウィジェットが一切描画されない」を Day3 に踏む形になっている。これは締切5日で最も回収コストの高い種類の手戻り。
- どう直すか: 実装最初の1時間を「Hello Widget」（静的な1枚カードが ChatGPT inline に出るだけ）の疎通に充て、リソース登録の正解形を確定してから analyze_dog_signal に進む。ビルド順序 §9 の「1. 基盤」にこれを明記する。

### Major

**M-1. 二重LLMアーキテクチャが「GPTラッパー」批判をむしろ強める構図になっている**
- 場所: requirements.md §6.4（入力＝「音声の記述」→ MCPサーバが GPT-5.6 を呼ぶ）
- なぜ危険か: 音声の「記述」を作るのは ChatGPT（=GPT-5.6）であり、それを自サーバから**同じモデル**に送り直して4点カードを得る設計は、審査員に「ChatGPT が既にできることをAPI課金して往復しているだけ」と一撃で要約されうる。技術実装軸の防御線（スキーマ強制・研究コーパス注入・履歴統合・確信度キャリブレーション）は PRD に書かれているが、**それらが出力物として審査員の目に見える要件になっていない**。見えない付加価値は無いのと同じ。
- どう直すか: (a) カードのフッターに根拠フィールド（どの研究知見・どの確認済み観察を使ったか）を表示する要件を FR に昇格、(b) README に「GPT-5.6 が生成するもの／アプリが強制・付加するもの」の境界図（§6.4 を図化）＋ **eval 表（8.1 の9ケース×AC適合の実測結果）**を載せる。9ケースの実行ログはラッパー批判への最強の反証であり、5日でも作れる。

**M-2. スコープの自己矛盾: 「来客以外は対象外」なのに、7状況セレクタ・テストケース・スライドが来客以外（しかもMVP外の動画入力）を約束している**
- 場所: §3.2 non-goal「来客以外の状況を同等の品質で解釈すること」「短尺動画解析」 vs FR-2「写真/**短い動画**を追加入力できる」・FR-3「7区分から選択」・§8.1 ケース2「留守番前＋クンクン鳴き＋**動画**」ケース3「遊び」ケース4「食事」・スライド4「音声＋写真/**動画**」・スライド5の7状況リスト
- なぜ危険か: 審査員は UI に「留守番」「散歩」の選択肢があれば必ず試す。そこで品質が崩れれば「来客だけ作り込んだデモ用アプリ」と映り、non-goal の但し書きは読まれない。動画に至っては、MVP外と宣言したものを FR とテストケースが要求しており、実装者がどちらに従うべきか決定不能。
- どう直すか: **動画への言及を FR-2・§8.1ケース2・スライド4/5 から全削除**（将来像の節のみに残す）。来客以外の状況が選ばれた場合の挙動を定義する: 低確信度で応答した上で「この場面はまだ来客ほど較正されていません」と明示する“誠実な縮退”は、むしろ「不確実性の通訳」というブランドの実演になる。矛盾を1時間で武器に変えられる稀な指摘。

**M-3. 「済み」と記録された未完了作業が既に発生している — デッキに英語フックは無く、「デモの顔」はプロトタイプに存在しない**
- 事実確認:
  - review-pawlens-winnability.md §2 は「**スライド1・10も英語フック併記済み**」と記載。しかし pptx の inspect.ndjson を全走査した結果、スライド1は「犬の『うるさい』を、観察できるサインへ。」等**全て日本語**、スライド10も同様。英語はヘッダの定型文のみ。**「済み」は虚偽**。
  - design-spec §10「断定回避リビール」は「3分デモの最大の見せ場」「デモの顔」と位置づけられているが、プロトタイプ bundle をデコードして検証した結果、ghost / 取り消し線 / 「決めつけません」/ "We don't put words" は**0ヒット＝完全未実装**（霧インジケーターと波形は実装済み）。さらに spec §4/§5 が必須とする `prefers-reduced-motion` 対応も 0 ヒット。
- なぜ危険か: 個々の未実装より、**進捗台帳が現実より先に「完了」を打つ癖**が危険。この癖が evidence ledger（Codex Session ID・デモURL）に波及すると、提出直前に「あるはずのものが無い」が連鎖する。またデモ台本 0:20–0:40 は存在しない演出を前提に組まれており、動画制作が Day5 に破綻する。
- どう直すか: 今日、evidence ledger を「実物へのリンクが貼れるまで未完了」ルールで運用開始。リビールは**今日デスコープ判断**: §10.2 に既に定義されている reduced-motion 用の静的 before→after 表現（「よくある翻訳: ~~こわいよ！~~ → PawLensの見立て」）を正式版に昇格させれば、思想は伝わり実装コストは1/10になる。デッキは動画字幕に注力し、スライド英語化は表紙＋3/6/9枚目の併記に留める。

**M-4. TAM算定にカテゴリ錯誤があり、精査されると数字が崩れる**
- 場所: research-pawlens.md §1.6「米6,500万世帯 + **欧州1.4億世帯(犬飼育世帯)** + 日本550万世帯 ≈ 2.1億世帯 × $15 ≈ $3.2B」
- なぜ危険か: 同じ文書の §1.4 が「欧州140百万世帯は**ペット全体**の飼育世帯。犬は**世帯の25%**」と正しく書いている。欧州の総世帯数 ≈ 140M÷0.49 ≈ 286M、その25% ≈ **約7,100万世帯**が犬飼育世帯。正しいTAMは約1.4億世帯 ≈ **$2.1B** で、記載値は約1.5倍の過大。文書自身が「都合の良い数字を選んでいると見抜かれるリスク」を警告しながら、自らその状態にある。審査員が電卓を叩く確率は低いが、叩かれたら Impact 軸の信頼が丸ごと落ちる。
- どう直すか: 5分で修正可能。$2.1B に直すか、提出資料からTAM金額を落として「99.12% / 69.3% / 吠え1位」の一次研究3点に絞る（後者を推奨。金額TAMはこのプロダクトの説得構造に不要）。

**M-5. デモ締めコピー "99% of dogs have a behavior problem — and most of us never notice" は出典の範囲を超えた合成主張**
- 場所: positioning-uplift §6（2:40–3:00）・§1.3
- なぜ危険か: 69.3% は「**手放した飼い主**のうち問題を認識していなかった割合」であり、"most of us（一般飼い主）never notice" への一般化は一次研究が支持しない（research 自身が一般飼い主の非認識は34.5%と記録している＝過半数ではない）。「捏造のない引用可能な一次情報」を旗印にしたプロジェクトが、最後の10秒で旗印を裏切る構図。
- どう直すか: "…and 7 in 10 owners who gave up their dog never saw it coming." のように主語を出典どおり手放した飼い主に限定する。強度はほぼ落ちない。
