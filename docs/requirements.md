## 0. サマリー

**PawLensは、犬の鳴き声・写真・状況・その子の確認済み観察履歴を手がかりに、「今この犬は何を必要としている可能性があるか」についての根拠付き仮説と、安全な次の一手を返すChatGPTアプリ。**

犬語を人間語へ断定的に「翻訳」するのではなく、**不確実性を明示したまま、人間の観察と共感を拡張する通訳**である。これが既存の「犬語翻訳アプリ」との決定的な差別化点。

最終的に実現させたいことは、**犬がすでに発しているサインを、人が見落とさず理解できる社会にする(犬ともっと分かり合える社会を作る)**こと。PawLensは一度の推定で終わらず、飼い主が確認した事実を次回の理解へつなぐ。

- ワンライナー: *PawLens helps people hear beyond a bark—turning a dog's voice, context, and history into research-grounded observations that foster safer, more empathetic everyday decisions.*
- 中核の逆張り: 「断定翻訳」ではなく **「不確実性の通訳」**。決めつけを観察へ変える。
- 差別化の一文: *PawLens does not translate dogs. It turns uncertainty into better observation over time.*
- プロダクト約束: AIの仮説と飼い主が確認した事実を混同せず、後者だけを「その子の手帳」として蓄積する。

## 1. Problem / Why now / なぜこの形態か

### 1.1 Problem
犬は鳴き声・視線・姿勢・行動の変化で状態を伝えているが、多くの人はそれを「うるさい」「しつけの問題」として片付ける。飼い主でさえ、来客・留守番前・散歩中の反応を前に「不安か興奮か要求か分からない」「何を観察し、どう対応すべきか分からない」という課題を抱える。犬を社会の一員として受け入れるには、制度以前に**誰もが犬の状態を想像し、配慮ある行動を選べること**が要る。

### 1.2 Why now
- GPT-5.6 が音声・画像・会話・履歴を横断して扱えるようになった。
- 一方、鳴き声を人間語へ1対1で「翻訳」する科学的根拠はまだ不十分（後述の参考文献）。
- だから今必要なのは、もっともらしい断定翻訳ではなく、**研究知見＋個体・状況の文脈を踏まえて人間が考えるための支援**。この設計思想が「今」正しい。

### 1.3 なぜ ChatGPT App 形態か
- 大会の「agent plugins（skills, MCPs, tools）」枠に明確に合致し、GPT-5.6利用が自然かつ中核になる。
- 飼い主は ChatGPT の会話内で音声・写真を添付でき、モデルが文脈を保持したまま履歴・追質問に対応できる。
- 審査員が ChatGPT Developer Mode 経由で試せる（judge-testable）。

### 1.4 個人の原体験（応募資料での語り方）
「ヨーロッパは犬に優しく日本は遅れている」と一般化せず、個人の気づきとして語る:
> I realized that the way a society makes room for dogs changes how people notice their needs. I wanted to make canine understanding more accessible—starting with the small moments people usually overlook.

## 2. ターゲットユーザー / ジョブストーリー

### 2.1 最初のターゲット（絞る）
- 日本の都市部で暮らす犬の飼い主
- 吠え・留守番・来客・散歩前後など、日常の小さな不安を抱える人
- 家族や友人にその子を理解してもらいたい飼い主

「世界中の犬」は将来像。まず狭く深く、観察履歴と信頼を蓄積する。

### 2.2 主要ジョブストーリー
> 犬が来客時に激しく吠えたとき、私は「これは警戒か興奮か、今どうすべきか」を落ち着いて判断したい。だから、鳴き声と状況と写真を渡すだけで、根拠付きの見立てと安全な次の一手が欲しい。

### 2.3 測定可能な「成功の瞬間」
飼い主が提示された「観察ポイント」を実際に確認し、**決めつけ（"うるさい"）ではなく観察（"耳が後ろ、尻尾が下がっている＝警戒かも"）に基づいて次の行動を選べた**こと。次回には、確認済みの観察を根拠として前回との差分を一つ理解できること。

### 2.4 アウトカム仮説（Impactの検証対象）
来客時の吠えに迷う飼い主がPawLensを使うと、単なる感情ラベルではなく「次に観察するサイン」と「安全な次の行動」を選べるようになる。その結果、叱責や不用意な接近ではなく、刺激を下げ、犬に距離と選択肢を与える対応を取りやすくなる。

この仮説の検証指標はモデルの感情推定正解率ではない。以下を測定する。

- 出力後に飼い主が観察すべき身体サインを2つ以上説明できる。
- 飼い主が安全な次行動を一つ選び、その理由を説明できる。
- 2回目の記録で、前回との差分と根拠を区別できる。

## 3. スコープ

### 3.1 MVPに含む（来客時の吠えに絞る）
- 犬の鳴き声（〜10秒の音声）の入力
- 任意の写真（姿勢・耳・尻尾・視線の視覚手がかり）の入力。短い動画はMVP外。
- 状況は **来客/チャイム** を主シナリオとし、補足として「直前の出来事」「人との距離」を入力
- 最小の個体プロフィール（名前・任意の性格メモ）
- 「今の見立て」カード（後述の4点）の生成
- 飼い主が実際に見えたサインの確認・記録（観察ログ）
- 2回目の記録で、**確認済みの観察だけ**を根拠にした前回との差分表示（履歴保証は**同一会話内**の2ラウンドまで。会話を跨ぐ再訪はベストエフォート＝§6.3の個体同定を参照）

### 3.2 Non-goals（明示的に対象外）
- 鳴き声から人間のセリフへの断定的翻訳（そもそも作らない。設計思想に反する）
- 医療診断・行動学的診断の断定（獣医・行動専門家へ誘導する）
- 首輪/ハーネス型センサー（心拍・HRV・皮膚温）＝**将来像として語り、MVPからは除外**（ハードウェア依存でjudge-testable性が下がるため）
- リアルタイム連続モニタリング
- 来客以外の状況を同等の品質で解釈すること
- 短尺動画解析、複数犬の管理、詳細なプロフィール管理
- 多言語展開（MVPは日本語中心＋英語の応募資料）

### 3.3 将来像（応募のWhy nowを支える"深さ"の証明として言及）
音声＋動画姿勢＋心拍/HRVのマルチモーダル統合（Garcia-Loya 2025 らの研究設計に接続）。MVPは実装せず、ロードマップとして提示。

## 4. ユーザーフロー

```
トリガー: 犬が吠えた / 落ち着かない
  ↓
① 入力・同意: ChatGPT会話で音声＋（任意で写真）を添付し、状況を選択
  ↓
② モデル処理: MCPツール analyze_dog_signal が GPT-5.6 へ
   「音声の記述＋視覚手がかり＋来客文脈＋個体プロフィール＋確認済み履歴＋研究知見」を渡し、
   構造化出力（JSON）で状態仮説を生成
  ↓
③ レビュー: ウィジェットが「今の見立て」カードを表示
   （状態仮説 / 確信度と限界 / 観察ポイント / 次の一手）。仮説の輪郭は確信度に応じて「霧が晴れる」ように表示するが、ラベルと限界文も必ず併記する
  ↓
④ 確認とアクション: 飼い主が「実際に見えたサイン」をチェックし、主CTAで安全な次の一手を実行・記録する（save_observation）
  ↓
⑤ 観察可能な結果: 記録が履歴に蓄積し、次回は「その子にとって普段と違う変化」を、AI仮説ではなく確認済み観察を根拠に反映する
```

## 5. 「今の見立て」カード＝出力仕様（プロダクトの核）

GPT-5.6 は必ず以下4点を構造化出力で返す。**セリフ（犬の一人称）は返さない。**

| # | 項目 | 内容 | 例 |
|---|---|---|---|
| 1 | 状態仮説 | 警戒/不安/興奮/遊びへの期待/要求 等の可能性（複数可・確率的表現） | 「来客直後の文脈から**警戒**の可能性が中程度。次点で興奮」 |
| 2 | 確信度と限界 | 確信度（低/中/高）＋なぜ断定できないか | 「音声と1枚の写真だけでは断定できません」 |
| 3 | 観察ポイント | 今すぐ確認すべき身体サイン | 「耳の向き・尻尾の高さ・視線・体の硬さ・直後の行動」 |
| 4 | 次の一手 | 安全で低刺激な対応。悪化時は専門家へ | 「距離を取り刺激を下げる。変化が続けば獣医/行動専門家へ相談」 |

補助要素:
- **不確実性が高い場合の追加質問**（例: 「吠える前に何がありましたか？」）
- **履歴比較**（例: 「前回の来客時より落ち着いています」）
- **緊急性フラグ**（うずくまり・過度の震え・攻撃兆候等が疑われる場合の安全案内）

### 5.1 仮説と確認済み観察の分離（必須）

出力・保存・履歴表示では、以下の二つを視覚的にもデータ構造上も混在させない。

| 種別 | 作成者 | 保存・利用ルール |
|---|---|---|
| AIの仮説 | GPT-5.6 | 今回の見立てとして表示する。根拠・限界・確信度を必須とし、次回の「事実」として扱わない。 |
| 確認済み観察 | 飼い主 | 飼い主がチェックまたは自由記述で確認したものだけを保存する。次回比較の根拠に使える。 |

この分離により、PawLensは一般論を繰り返すのではなく、飼い主と共に「その子の手帳」を育てる。

## 6. アーキテクチャ（ChatGPT Apps SDK + MCP）

> ⚠️ 実装前に **OpenAI Apps SDK docs を `$openai-docs` で必ず再確認**すること（`apps-sdk/build/mcp-server`, `apps-sdk/build/chatgpt-ui`, `apps-sdk/plan/tools`, `apps-sdk/reference`）。本節はドラフト。
> ⚠️ **C-4 警告**: 本節の `_meta.ui.resourceUri` / MIME `text/html;profile=mcp-app` / `ui/notifications/tool-result` は**汎用 MCP Apps（SEP系）の規約**であり、OpenAI Apps SDK の実仕様（`ui://` リソース＋`openai/outputTemplate` メタデータ等）と体系が異なる可能性がある。**実装最初の1時間で「Hello Widget」（静的1枚カードの inline 描画）を疎通させ、リソース登録の正解形を確定してから** `analyze_dog_signal` に進むこと（§9 ビルド順序 1）。

### 6.1 アプリのアーキタイプ
**react-widget（decoupled data/render パターン）**。データツールと描画ツールを分離し、ウィジェットが毎回再描画されないようにする。

### 6.2 ツール設計（1ツール1ジョブ）

| ツール名 | 種別 | annotations | 役割 |
|---|---|---|---|
| `analyze_dog_signal` | data + render | `readOnlyHint: true`, `openWorldHint: false` | 音声/画像の記述＋来客文脈＋プロフィール＋確認済み履歴を受け取り、GPT-5.6で4点カードを生成。`structuredContent` に仮説カードを返し、`_meta.ui.resourceUri` でカードウィジェットを描画 |
| `save_observation` | mutating | `readOnlyHint: false`, `idempotentHint: false` | 飼い主が確認した実際のサイン、実行した次の一手、時刻だけを履歴へ保存。AIの仮説は事実として保存しない。 |
| `get_dog_history` | data | `readOnlyHint: true` | その子の確認済み観察ログを取得し、「前回との差分」の根拠に使う |
| `manage_dog_profile` | mutating | `readOnlyHint: false`, `idempotentHint: true` | 個体プロフィールの作成・更新・**削除**（削除時は紐づく観察ログも消去。§10「飼い主が削除可能」の実装点） |

### 6.3 コンポーネント
> 具体的な技術選定は **6A. 技術スタック** を参照（Cloudflare Workers + Hono + `@hono/mcp` + `@modelcontextprotocol/sdk`）。
- **MCP サーバ**（TypeScript / Hono + `@hono/mcp`, Cloudflare Workers）: `/mcp` エンドポイントで Streamable HTTP を処理。`@modelcontextprotocol/sdk` の `McpServer` でツール登録・ウィジェットリソース登録（MIME `text/html;profile=mcp-app`）。
- **ウィジェットUI**（React + Vite, Workers Static Assets 配信）: MCP Apps bridge（`ui/notifications/tool-result`）で `structuredContent` を受け取りカード描画。フォローアップは `window.openai.sendFollowUpMessage` / `callTool`。
- **ストレージ**: 個体プロフィールと観察ログ。MVPは **Workers KV**（D1移行は提出後）。個体は匿名IDで管理。
- **個体同定（履歴機能の鍵・C-2対応）**: 認証なしMVPでは会話を跨いで同一IDへ確実に辿る手段がないため、**履歴の保証範囲は同一会話内**とする。`manage_dog_profile` が発行する `dog_id` を `structuredContent` と `widgetState` の両方へ埋め、以降のツール呼び出し引数に `dog_id` を含めるようツール記述子で誘導する。judge test card には「**同じ会話内で2回試す**」手順を明記し、会話跨ぎの再訪は「動けばボーナス」扱いとする。
- **研究コーパス**: 後述の論文知見を要約したプロンプト内リファレンス（RAG的に該当状況の知見のみ注入し、トークン最適化）。

### 6.4 GPT-5.6 のプロダクト境界（README/デモで可視化する中核）
- 入力: 音声の記述（or 文字起こし不可な非言語音の特徴）＋視覚手がかり＋来客文脈＋個体プロフィール＋**確認済み**直近履歴＋研究知見サマリ
- 出力: **厳格なJSONスキーマ**（状態仮説[]／根拠[]／確信度／限界文／観察ポイント[]／次の一手[]／追加質問?／緊急性フラグ）
- ガードレール: セリフ生成の禁止、確信度・限界の必須化、医療/行動診断の断定禁止、緊急時の専門家誘導を必須フィールド化。
- 信頼性: スキーマバリデーション、確信度が低い時は追質問へフォールバック、リトライ。

### 6.5 セキュリティ / CSP / シークレット
- `_meta.ui.csp` に `connectDomains` / `resourceDomains` を厳密指定。
- APIキーは環境変数（`.env.example` に値なしで記載）、ソース管理に含めない。Cloudflare では `wrangler secret put` でシークレットを管理。
- 個体・観察データは最小限・匿名IDで保持。飼い主が削除可能。

### 6A.2 トランスポートと構成方針
- **トランスポートは Streamable HTTP**（SSE ではなく HTTP ベース）。`@hono/mcp` の標準。クライアント（ChatGPT Developer Mode）は `https://<worker>.workers.dev/mcp` に接続。
- **MVPはステートレス構成**。状態は KV に `dog_id` キーで永続化し、ツール呼び出しは冪等に保つ。**履歴の保証範囲は同一会話内**（§6.3 個体同定を参照）。
  - 将来、セッション内のリアルタイム同期が必要になった場合のみ Cloudflare **Durable Objects / McpAgent** への移行を検討（MVPでは不要）。
- **ウィジェット配信**: Vite ビルド成果物を Workers Static Assets（または R2）で配信し、MCPリソースの `resourceUri` から参照。CSP の `resourceDomains` に配信ドメインを登録。

### 6A.3 OpenAPI の位置づけ
- MCP 本体は JSON-RPC（`/mcp`）で動くため OpenAPI の対象外。
- OpenAPI は **補助的な HTTP API**（個体プロフィールCRUD・履歴取得・ヘルスチェック）と、**ツール入出力スキーマの外部向けドキュメント**に用いる。
- Zod を SSOT とし、`@hono/zod-openapi` で「ツール入力検証」と「OpenAPI 仕様生成」を同一スキーマから供給（二重定義を避けトークン/保守コスト削減）。

### 6A.6 スタック採用に伴う留意点（技術リスク補足）
- **R-5 Apps SDK 例との差分**: 公式例は Node/Express or Python + Vite 構成。本プロジェクトは Cloudflare Workers + Hono へ置き換えるため、ウィジェットのリソース登録・CSP・静的配信は公式例を Workers 向けに翻訳する必要がある。実装前に `$openai-docs`（`apps-sdk/build/mcp-server`, `apps-sdk/build/chatgpt-ui`）と上記参考リポジトリを突き合わせる。
- **R-6 音声/画像の受け渡し**: Workers 上での添付ファイル（音声・画像）の受領経路と GPT-5.6 への引き渡し方式を検証（R-1/R-2 と連動）。
- **R-7 コールドスタート/実行時間**: GPT-5.6 呼び出しを含むため Workers の実行時間・サブリクエスト制約を確認。長い処理はストリーミング応答で体感を改善。

## 6B. UI/UX 要件

> **完全版のデザイン仕様書は [`docs/ui_design/pawlens-widget-design-spec.md`](./ui_design/pawlens-widget-design-spec.md)**（カラーHEX・CSS変数・コンポーネント構造・4状態・モーション・アクセシビリティを網羅）。本節はその要点と、審査「デザイン」軸で満たすべき要件を抜粋する。

### 6B.1 デザインコンセプト
**「双眼鏡のような通訳 — 言い切らずに、観察の解像度を上げる」**。参照点は動物病院の問診票やAIチャット吹き出しではなく、**フィールドノート（観察記録）× 天気予報の確率表現**。「診断する装置」ではなく「観察者の手帳に知見が書き足される装置」として設計する。

### 6B.2 デザイン原則（5つ）
1. **言い切らない、けど頼れる（Confident Humility）** — 確信度のない断定表現をUIに存在させない。
2. **観察者の手帳（Field Notebook, not Diagnosis Sheet）** — 診断結果票でなく「その子について今わかっていることのメモ」の佇まい。
3. **静かな魔法（Quiet Magic）** — loading→success の1回の遷移にだけ「理解が像を結ぶ」演出を集中投下。常時アニメーションはしない。
4. **その子だけの記録（This Dog, Not Dogs in General）** — 名前・写真・履歴差分を必ず視界に入れ、一般論テンプレに見せない。
5. **安心への抜け道は常にある（Always an Exit to Safety）** — 緊急性フラグと専門家導線は、どの状態・表示モードでも1タップ以内。

### 6B.3 ビジュアル言語（要点）
- **カラー**: 「AI＝紫」の慣用句を排除。主色 **セージグリーン**（観察・落ち着き）× アクセント **テラコッタ**（犬の毛色・温かみ）。純白/純黒は使わず生成りの紙色・ウォームブラック。緊急性の赤はアクセントと色相を明確に分離。ライト/ダーク両テーマのCSS変数を定義済み。
- **タイポグラフィ**: self-contained制約下で、**見出し＝和文明朝／本文＝和文ゴシック**のシステムフォント組み合わせでコントラストを作る（Inter等の没個性スタックを避ける）。
- **確信度の視覚化（最重要）**: 進捗バー（0-100%塗り）は断定的に見えるため**禁止**。独自の**「霧が晴れるインジケーター」**（低確信度＝輪郭がぼやけ、高確信度＝澄む）＋必須の限界文で表現する。霧の演出は単なる装飾ではなく、AIの仮説を飼い主の確認によって見直す体験の起点とする。

### 6B.4 UI状態要件（4状態すべて必須）
- **loading**: 「魔法が起きている」演出。送信音声のダミー波形 → 確信度インジケーターの円へ収束するモーフィングを、応答到着＝「像が結ばれる瞬間」に同期。誇張（キラキラ/紫グロー）はしない。
- **empty**: フィールドノートの「白紙ページ」。破線枠でまだ何も書かれていないことを示し、プロフィール未登録なら名前登録へ誘導。
- **error**: システム不具合を緊急性フラグと明確に区別。原因推定＋次の行動を必ず添える。**部分的成功表示**（取れた情報だけ低確信度で出す）を優先し全滅表示を禁止。
- **success**: 見立てカードを描画。出現時のみセクション単位の1回限りstagger（400ms以内）。主CTAは常に一つだけにし、「今すぐできる安全な次の一手」を実行・記録へつなげる。

### 6B.5 見立てカードの構造要件
CardHeader（緊急性帯／名前・サムネ・時刻）→ **AIの仮説**（大きく1つ＋根拠＋確信度）→ 次点仮説（小チップ、3列均等にしない非対称）→ 限界文（必須・空欄禁止）→ **飼い主が確認する観察**チェックリスト（全て未チェック開始・`widgetState`保存）→ 次の一手（主CTA1つ＋専門家導線常設）→ 追加質問（不確実性が高い時）→ **確認済み観察に基づく**履歴差分（フッター）。

### 6B.6 表示モード要件（inline / pip / fullscreen）
inlineを主戦場とし**inlineで完結**する情報密度に設計。fullscreenは「深掘り」（次点仮説全件・観察全項目＋解説・履歴タイムライン）。**自動昇格はせず**、ユーザー操作（「もっと見る」）でのみ `requestDisplayMode` を呼ぶ。pipは主仮説＋CTAのみ。

### 6B.7 親しみ・コピー原則
- 犬は必ず**個体名**で呼ぶ（「あなたの犬」等の一般化禁止）。
- 語尾は断定しない（「警戒している可能性があります」）。**犬のセリフを作らない**（中核思想への違反、最優先禁止）。
- 限界は理由を伴って明示。次の行動を必ず具体で添える。二人称で語りかける。
- 使い続けるほど「その子の手帳が育つ」感覚（履歴の蓄積表示）。

### 6B.8 アクセシビリティ要件（抜粋・詳細は仕様書8章）
`:focus-visible` の可視フォーカスリング／タップターゲット44px以上／`aria-live`でloading進行を通知／写真alt・装飾SVGは`aria-hidden`／本文コントラスト4.5:1以上／色だけに依存せず確信度はラベル（低/中/高）併記／`prefers-reduced-motion`で代替表現に切替／`window.openai.theme`をリロードなしで追従。

### 6B.9 やってはいけないこと（アンチパターン抜粋）
犬のセリフ化・断定翻訳／確信度を進捗バーで表現／「AI＝紫」グラデーション／均等3列カードグリッドの反復／`animate-pulse`/`animate-bounce`・`transition: all`／loading以外の常時アニメーション／4状態のいずれかの実装省略（特にerrorの部分的成功表示）。

## 7. 機能要件・受け入れ基準・エッジケース

### 7.1 機能要件（FR）
- FR-1: 音声（〜10秒）を入力として受け取り処理できる。
- FR-2: 任意で写真を追加入力できる（動画は対象外。§3.3 将来像でのみ言及）。
- FR-3: 状況を7区分から選択（不明を含む）できる。
- FR-4: 4点カード（仮説/確信度・限界/観察ポイント/次の一手）を構造化出力で返す。
- FR-5: 出力にセリフ（犬の一人称）を含めない。
- FR-6: 確信度と限界を必ず表示する。
- FR-7: 緊急性が疑われる場合、専門家誘導の安全案内を表示する。
- FR-8: 飼い主が確認した観察と実行した次の一手だけを保存し、履歴として次回に反映する。
- FR-9: 個体プロフィールを作成・更新できる。
- FR-10: AIの仮説と、飼い主が確認済みの観察を別のラベル・データ項目・表示領域で扱う。
- FR-11: 低確信度の仮説を「霧が晴れる」インジケーターとラベルで表示し、飼い主の確認を促す。色やモーションだけに意味を依存しない。
- FR-12: 来客時の主CTAは安全で低刺激な行動を一つだけ提示し、実行・記録へつなげる。
- FR-13: カードの全テキスト（仮説ラベル・限界文・観察ポイント・次の一手・追加質問）は**リクエスト言語に追従**する（最低 en/ja。GPT-5.6への出力スキーマに `locale` を渡して実現）。
- FR-14: カードのフッターに**根拠フィールド**（どの研究知見・どの確認済み観察を使ったか）を表示する（「GPTラッパー」に見せないための可視化要件）。
- FR-15: 来客以外の状況が選択された場合、低確信度で応答した上で「この場面はまだ来客ほど較正されていません」と明示する（**誠実な縮退**＝「不確実性の通訳」思想の実演）。

### 7.2 受け入れ基準（AC）
- AC-1: 「来客時の吠え＋写真＋来客状況」を渡すと、警戒/興奮を含む複数仮説・確信度・観察ポイント・次の一手が返る。
- AC-2: 出力JSONがスキーマに適合し、`限界` フィールドが空でない。
- AC-3: セリフ調（「ぼく、こわいよ！」等）の文が出力に含まれない。
- AC-4: 2回目以降、同じ個体の履歴があると出力に「前回との差分」への言及が入る。
- AC-5: 情報不足時（音声のみ・状況=不明）は断定せず、追加質問を返す。
- AC-6: 飼い主が観察ポイントを一つ以上チェックして保存した場合、保存データはAI仮説を含まず、確認済み観察・時刻・実行した次の一手を持つ。
- AC-7: 同一個体の2回目の来客記録では、前回比較の根拠が確認済み観察としてUIに表示される。比較根拠がない場合は「比較できる記録がまだありません」と表示する。
- AC-8: 主要な来客フローでは、低確信度であっても主CTAが一つに絞られ、飼い主が安全な次の一手を選択・記録できる。
- AC-9: 霧のインジケーターは `prefers-reduced-motion` 環境で静的表現へ切り替わり、確信度ラベルと限界文は常に読める。
- AC-10: 英語で同等の入力を渡すと、カードの全テキスト（仮説ラベル・限界文・観察ポイント・次の一手・追加質問）が英語で返る（FR-13）。
- AC-11: 来客以外の状況を選択した場合、低確信度の応答に「まだ来客ほど較正されていない」旨の縮退文言が含まれる（FR-15）。

### 7.3 失敗・エッジ状態
- 音声が無音/短すぎ → 再入力を促す（空状態）。
- 画像に犬が写っていない → 視覚手がかりなしとして音声・状況のみで低確信度出力。
- モデル出力がスキーマ非適合 → バリデーション失敗として1回リトライ、それでも失敗ならエラーカード。
- 緊急兆候の記述 → 最優先で安全案内。

## 8. 評価計画（技術実装スコアの証拠）

### 8.1 代表テストケース（5〜10件）
1. 来客チャイム＋激しい吠え＋写真（耳前傾） → 警戒/興奮の複数仮説・中確信度
2. 留守番前＋クンクン鳴き（来客以外の状況を選択） → 低確信度＋「まだ来客ほど較正されていない」旨の誠実な縮退文言＋観察ポイント（FR-15）
3. 遊び中＋高い声＋写真（プレイバウ） → 遊びへの期待・ポジティブ
4. 食事前＋要求吠え → 要求の可能性
5. 状況=不明＋音声のみ → 断定せず追加質問（フォールバック確認）
6. 画像に犬なし → 低確信度で継続
7. 攻撃兆候の記述 → 緊急安全案内
8. 同一個体の2回目 → **確認済み観察**を根拠にした履歴差分への言及
9. 低確信度＋観察チェック → 霧のインジケーター、追加質問、主CTA一つが表示され、確認済み観察だけが保存される
10. **音声反証テスト**: 対照的な2音声（高く連続する興奮様の鳴き vs 低い唸り）＋同一状況 → 出力の仮説・観察ポイントが有意に異なる（音声入力が"飾り"でないことの実測証拠。変わらない場合は音声を入力契約から外し「飼い主の記述の構造化」へ正直に転換する）

### 8.2 「良い」の定義
- 断定を避け確信度・限界を明示している
- 観察ポイントが具体的な身体部位に落ちている
- 次の一手が安全・低刺激で、悪化時の専門家誘導がある
- 研究知見と矛盾しない（例: 短い音＝よりポジティブ傾向 等）
- AIの仮説、根拠、限界と、飼い主が確認した事実が混ざらない
- 履歴比較は推測ではなく、前回・今回の確認済み観察を根拠として説明される

### 8.3 手動テスト計画
MCP Inspector でツール記述子とウィジェット描画を確認 → ChatGPT Developer Mode（HTTPSトンネル）でエンドツーエンド → 上記ケースを手動実行し、AC を突き合わせ。

### 8.4 インパクト検証（小規模ユーザーテスト）※提出前は実施しない

> **2026-07-16 デスコープ判断**: 残り期間では実装と両立不能のため、**提出までは実施しない**（余力があれば知人2〜3人の非同期反応メモまで）。以下は提出後の検証計画として保持する。「未実施の数値を成果として応募資料へ書かない」という既存規律は維持。

対象は来客吠えに迷った経験のある飼い主5人程度とする。プロトタイプまたは動作版を用い、初回分析と2回目比較のシナリオを試してもらう。

| 検証項目 | 目標 | 記録方法 |
|---|---:|---|
| 観察サインの理解 | 5人中4人以上が、出力後に確認すべきサインを2つ言える | テスト直後の口頭回答 |
| 安全な行動選択 | 5人中4人以上が、選んだ次の一手と理由を説明できる | タスク完了と口頭回答 |
| 仮説と事実の区別 | 全参加者が、AIの見立てと自分の確認済み観察を区別できる | 画面上の指差し・質問 |
| 継続価値 | 5人中3人以上が、次回比較のために記録を残したいと答える | インタビュー |

これらは提出時に初めて結果を記載する。未実施の数値を成果として応募資料へ書かない。

## 9. ビルド順序（デモリスク順）

1. **基盤**: MCPサーバ雛形 + `/mcp` 疎通 + **Hello Widget**（静的カードが ChatGPT inline に描画されるまでを最初の1時間で確認し、リソース登録の正解形を確定＝C-4対策）+ **音声経路スパイク**（R-1/R-2 の白黒判定 → 入力契約とワンライナーを確定＝C-1対策）
2. **クリティカルハッピーパス**: `analyze_dog_signal` の構造化出力 + カードUI（来客ケースが動く）
3. **信頼性**: スキーマバリデーション・確信度必須・緊急フラグ・フォールバック
4. **履歴**: `save_observation` / `get_dog_history` と差分反映
5. **ポリッシュ**: loading/empty/error/success 状態、日本語UI、アクセシビリティ
6. **提出物**: README（Built with Codex）、3分デモ、judge test card、evidence ledger

## 10. 信頼と安全（Trust / Safety）

- 断定翻訳をしない（設計原則）。すべて「可能性」「仮説」として提示。
- 医療・行動診断は断定せず、獣医師・行動専門家への相談を促す。
- 確信度と限界を常時可視化。情報不足時は追加質問。
- データは最小限・匿名ID・飼い主による削除可能。
- 出力の免責（教育・観察支援であり専門的判断の代替ではない旨）を明示。
- AIの仮説は確認済みの事実として保存・比較しない。履歴の根拠は飼い主が確認した観察に限定する。
- 音声・写真には人や住所などの個人情報が含まれ得るため、保存対象・保存期間・削除方法・第三者提供の有無をプライバシー説明に明記する。デモ素材は利用許諾済みのものだけを使う。

## 11. 参考文献コーパス（研究根拠）

> 結論: 鳴き声だけで「次の行動」を直接予測する研究は少なく、強い先行研究は主に「鳴き声 → 状況(文脈)・個体属性・人が知覚する感情」の推定。実用設計では音声＋動画/画像＋飼い主入力＋直前の出来事を合わせるのが妥当。→ PawLensの設計思想を裏付ける。

### 11.1 状況・感情・個体の推定（音声中心）
- **Hantke et al., 2018 (ICASSP)** — 音響特徴から文脈・知覚感情・感情強度を推定。文脈分類は人間水準を上回り得ると報告。**最重要先行研究**。
- Yin & McCowan, 2004 — 来客/隔離/遊びの3状況で音響的下位タイプと個体識別。
- Molnár et al., 2008 — 鳴き声の文脈特異性・個体特異性の基礎研究。
- Larrañaga et al., 2015 — 性別85%/年齢80%/7状況56%/個体68%を分類。**個体キャリブレーションの重要性**（＝履歴機能の根拠）。
- Gómez-Armenta et al., 2024 — 深層学習で個体/犬種/年齢/性別/文脈を分類。支援用途を明示。
- Abzaliev et al., 2024 — 人間音声の自己教師あり表現を犬の鳴き声へ転用しベースライン改善。
- Scheumann et al., 2014 — 人は犬の発声を快‐不快×覚醒度で評価。短い音＝よりポジティブ、高ピッチ＝高覚醒。
- Faragó et al., 2017 — うなり声の文脈・感情の人による推定（bark以外の扱いに有用）。
- Molnár et al., 2009 — 犬自身が「来訪者」と「単独時」の鳴き声を区別。

### 11.2 マルチモーダル（将来像の根拠）
- **Dang et al., 2025 (EmotionalCanines)** — 1,400 bark を覚醒度×快不快でラベル化。**動画文脈の重要性**と主観ラベルの限界を明示。
- Garcia-Loya et al., 2025 — カメラ＋加速度/ジャイロ＋心拍＋赤外線温度を同期。融合モデルが最良 F1 0.96。
- Katayama et al., 2016 — HRVが正負の状況差に反応。
- Franzini de Souza et al., 2018 — 花火恐怖を行動＋HRV＋コルチゾールで統合評価。
- Siniscalchi et al., 2018 — 人の感情音声への犬の生理・行動反応。
- Polgár et al., 2020 — 福祉評価は生理指標単独では不十分、行動観察と併用すべき（＝観察ポイント機能の根拠）。
