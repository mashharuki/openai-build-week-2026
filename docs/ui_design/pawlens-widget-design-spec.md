# PawLens ウィジェットUI デザイン仕様書

> 対象: ChatGPT App（Apps SDK）ウィジェット `analyze_dog_signal` の描画UI
> 参照実装スキル: `intentional-design-guard`（AIスロップ回避）/ `frontend-design`（美的方向性）/ `web-design-guidelines`（Vercel Web Interface Guidelines）

---

## 0. コンセプトの前提（実装前に必ず共有する3問）

1. **誰が、何のために使う画面か**: 日本都市部の犬の飼い主が、鳴き声・写真・状況を渡した数秒後に読む「見立てカード」。診断書ではなく、観察を助けるための一枚。
2. **ムード**: 「静けさ」と「誠実さ」を土台に、「発見の高揚感（魔法感）」を一瞬だけ差し込む。終始テンションを張らない。
3. **類似しない参照点**: 動物病院の問診票やAIチャットの吹き出しUIではなく、**フィールドノート（野帳・観察記録）× 天気予報の確率表現（降水確率60%のような、言い切らないが実用的な情報デザイン）**。PawLensは「診断する装置」ではなく「観察者の手帳に知見が書き足される装置」として設計する。

一言コンセプト: **「双眼鏡のような通訳 — 言い切らずに、観察の解像度を上げる」**

---

## 1. デザイン原則

1. **言い切らない、けど頼れる（Confident Humility）**
   すべての仮説には確信度が同伴する。確信度のない断定表現はUI上に一切存在させない（コピー原則は7章）。
2. **観察者の手帳（Field Notebook, not Diagnosis Sheet）**
   カードは「診断結果票」ではなく「その子について今わかっていることのメモ」に見える佇まいにする。罫線・紙質感（ノイズテクスチャ）・手書き的アクセントを控えめに使う。
3. **静かな魔法（Quiet Magic）**
   常時アニメーションではなく、loading→successの1回の遷移にだけ「理解が像を結ぶ」演出を集中投下する。他の箇所は静的で抑制的。
4. **その子だけの記録（This Dog, Not Dogs in General）**
   名前・写真・過去の履歴差分を必ず視界に入れ、一般論のテンプレートに見せない。
5. **安心への抜け道は常にある（Always an Exit to Safety）**
   緊急性フラグと専門家導線は、どの状態・どの表示モードでも1タップ以内に到達できる場所に固定する。

## 2. ビジュアル言語

### 2.1 カラーパレット

主色は紫を避け、**セージグリーン（観察・自然・落ち着き）**を主色、**テラコッタ（犬の毛色・温かみ）**をアクセントに採用。ニュートラルは純グレーではなく主色の色相を1〜3%混ぜたウォームグレーを使う。

```css
:root {
  /* ライトモード */
  --color-primary: #4B6357;        /* セージグリーン深め: ヘッダー、主仮説強調、進行インジケータ */
  --color-primary-soft: #E4EAE3;   /* セージの淡色: 選択状態の背景 */
  --color-accent: #C1622D;         /* テラコッタ: 次の一手CTA、ポジティブアクション */
  --color-accent-soft: #F3DFCB;    /* テラコッタ淡色: CTAのホバー背景 */
  --color-urgent: #B23A2E;         /* 緊急性フラグ専用。CTAのaccentとは別トーンで混同を防ぐ */
  --color-urgent-soft: #F6DEDA;

  --color-surface: #FAF7F2;        /* 背景: 生成りの紙色。純白 #FFFFFF は使わない */
  --color-surface-alt: #F1ECE3;    /* カード内セクション区切りの背景 */
  --color-surface-raised: #FFFFFF; /* カードそのものの地色は白寄りに1段階だけ持ち上げる */
  --color-border: #E4DDCF;

  --color-text: #2B2A26;           /* ウォームブラック。純黒 #000 は使わない */
  --color-text-muted: #6B675E;
  --color-text-faint: #9A968A;

  /* 確信度専用スケール（断定に見せないため彩度を抑え、明度のみで段階化） */
  --confidence-low: #C9BBA1;    /* にじんだ低彩度 */
  --confidence-mid: #8FA68F;
  --confidence-high: #3E5449;
}

:root[data-theme="dark"] {
  --color-primary: #8FB3A0;
  --color-primary-soft: #24352C;
  --color-accent: #E08A55;
  --color-accent-soft: #3A2A1E;
  --color-urgent: #E2695C;
  --color-urgent-soft: #3A2320;

  --color-surface: #1C1B18;
  --color-surface-alt: #26241F;
  --color-surface-raised: #232019;
  --color-border: #34312A;

  --color-text: #ECE7DC;
  --color-text-muted: #A8A296;
  --color-text-faint: #726E63;

  --confidence-low: #6B6152;
  --confidence-mid: #6E8F7C;
  --confidence-high: #9FCBB0;
}
```

**役割の要点**:
- `--color-accent`（テラコッタ）と `--color-urgent`（赤みの強いテラコッタ寄り赤）は隣接色にせず、彩度と色相を明確に離して視覚的に混同しないようにする。
- 「AI機能インジケータ＝紫」という慣用句を避けるため、loading演出でも紫は使わずセージ〜アンバーのグラデーションのみで表現する（4.1節）。
- Tailwind使用時は上記をそのまま `theme.extend.colors` に登録し、ユーティリティでのHEX直書きを禁止する。

### 2.2 タイポグラフィ

self-contained制約（外部フォント不可）に対応するため、**OS標準搭載の和文セリフ／和文ゴシックの組み合わせ**でコントラストを作る。Inter/Roboto/システムゴシック単体のような没個性スタックにしない。

```css
:root {
  --font-heading: "Yu Mincho", "YuMincho", "Hiragino Mincho ProN", "Noto Serif JP", serif;
  --font-body: "Yu Gothic", "YuGothic", "Hiragino Kaku Gothic ProN", "Noto Sans JP", -apple-system, sans-serif;
}
```

| 用途 | フォント | サイズ | ウェイト | 行間 |
|---|---|---|---|---|
| カード最上位見出し（犬の名前＋一言仮説） | heading（明朝） | 17px | 500 | 1.4 |
| セクションラベル（状態仮説／観察ポイント 等） | body（ゴシック） | 12px / letter-spacing 0.04em | 600 | 1.3 |
| 本文（仮説文・限界文） | body | 14px | 400 | 1.7 |
| 補助テキスト（タイムスタンプ・出典注記） | body | 12px | 400 | 1.5 |
| 数値（確信度%、時刻） | body + `font-variant-numeric: tabular-nums` | 13px | 500 | — |

- 見出し=明朝／本文=ゴシックの組み合わせで「セリフ×サンセリフ混合」条件を満たし、ウェイト差だけに頼らない。
- 1画面で使うフォントファミリーは2つ（heading/body）に限定。
- 見出しはinline幅で折り返しても破綻しないよう `text-wrap: balance` を指定。

### 2.3 角丸・余白・エレベーション

| トークン | 値 | 用途 |
|---|---|---|
| `--radius-card` | 16px | 見立てカード全体、写真サムネイル |
| `--radius-chip` | 999px | 確信度バッジ、状態仮説タグ、観察ポイントのアイコンバッジ |
| `--radius-control` | 10px | ボタン、入力欄 |
| `--space-unit` | 4px | 8の倍数リズム（4/8/12/16/20/24px）。inlineは外周16px、セクション間20px |

エレベーションは**2段階のみ**（Appleの「控えめさ」原則）:
- 静止時: `box-shadow: 0 1px 2px rgba(43,42,38,0.06); border: 1px solid var(--color-border);`
- フォーカス/ホバー時のみ: `box-shadow: 0 4px 16px rgba(43,42,38,0.10);`
強いドロップシャドウ（`shadow-lg`、`shadow-2xl`相当）や紫がかったグロー効果は使わない。カード背景には微細な紙のグレインテクスチャ（`background-image` に極薄ノイズSVGをdata URIでインライン、`opacity: 0.03`）を敷き、単色べた塗りを避ける。

## 3. 見立てカードのコンポーネント構造

```
┌─ CardHeader ─────────────────────────────┐
│ [緊急性フラグ帯]（該当時のみ、カード最上部に固定）        │
│ 🐾サムネ  むぎ・3歳・柴犬            18:42の記録        │
├─ PrimaryHypothesis（主仮説：大きく1つ）──────┤
│  "警戒" の可能性  ● 確信度：中          │
│  （霧が晴れるインジケーター）                    │
├─ SecondaryHypotheses（次点仮説：小チップ横並び）──┤
│  興奮 32%相当 ｜ 遊びへの期待 12%相当            │
├─ LimitationNote（限界文）───────────────┤
│  「音声10秒と静止画1枚のみのため、体の硬さの変化までは│
│    読み取れていません」                          │
├─ ObservationChecklist（観察ポイント）───────┤
│  ☐ 耳の向き    ☐ 尻尾の高さと振り方             │
│  ☐ 視線の先    ☐ 体の硬さ                     │
│  ☐ 直後の行動                                │
├─ NextStep（次の一手）─────────────────┤
│  [ 主CTA: 低刺激に距離を取って様子を見る ]         │
│  [ サブ: 経過を記録する ] [ サブ: 専門家に相談する ]│
├─ FollowUpPrompt（不確実性が高い時のみ）───────┤
│  "吠える前、来客はチャイムの前から見えていましたか？"│
└─ HistoryDiff（履歴がある場合、フッターに小さく）──┘
   前回（3日前）との違い：警戒の頻度がやや増加
```

### 3.1 状態仮説の複数併記（断定に見せない工夫）

- 主仮説1つを**大きく**、次点仮説は**小さなピル型チップ**で並べる（3列均等カードにしない＝非対称レイアウト）。
- 仮説の確率的表現は「62%」のようなにせの精密数値を主表示にしない。主仮説には `低 / 中 / 高` の3段階ラベルのみを大きく出し、次点仮説には相対的な太さの棒（幅で強弱を示す、数値ラベルは補助のみ・8px・text-faint）を添える。
- ラベル文言は必ず「〜の可能性」「〜かもしれません」で終わる（7.3節コピー原則と連動）。

### 3.2 確信度と限界の視覚表現（最重要・断定回避の核）

数値バッジや進捗バー（0-100%の塗りつぶし）は「診断の確信度」を客観的事実のように見せてしまうため**使わない**。代わりに独自の「霧が晴れるインジケーター」を使う。

- 実装: 直径28pxの円を3つ、わずかに重ねて配置。低確信度は円の輪郭がぼやけ（`filter: blur(3px)`、`opacity: 0.5`）、高確信度になるほど輪郭が澄む（blur 0、opacity 1）。色は `--confidence-low/mid/high` を使用。
- 隣に必ず1行の限界文（なぜ断定できないかの理由。空欄禁止、必須フィールド）を本文サイズで併記する。限界文はグレーアウトではなく通常の`--color-text-muted`で人間らしい文体にする（「AIの制約」ではなく「今わかっていること／わかっていないこと」というフレーミング）。

### 3.3 観察サインのチェックリスト

- アイコン+ラベルのリスト。アイコンはSVGインライン（線画・1.5px線幅・角丸端点で統一、絵文字は使わない）。
- チェックボックスは「AIが確認済み」ではなく「**あなたが次に確認する項目**」であることを明確にするため、初期状態は全て未チェックの`☐`。ユーザーがタップすると`☑`になり、`widgetState`にローカル保存（次回開いたときも保持、MCPへの送信は任意）。
- 各項目に12pxの短い補足（例: 耳＝「後ろに伏せていないか」）を添え、観察の着眼点を具体的にする。

### 3.4 次の一手のCTA

- 主CTAは1つのみ（`--color-accent`背景、白文字、`--radius-control`）。並列に3つのCTAを並べない。
- 悪化時の専門家導線はテキストリンク＋控えめなアイコンで常設（ボタン化しすぎて煽らない）。緊急性フラグが立っている場合のみ、専門家導線もCTA相当の視覚重みに昇格する。
- CTAタップ後は `window.openai.sendFollowUpMessage` で会話に「〜について記録しました」等のフォローアップを送る（詳細は6章）。

## 4. 4状態のUI仕様

### 4.1 loading（"魔法が起きている"待機演出）

**目的**: GPT-5.6の処理を「何もしていない静止」ではなく「観察が像を結びつつある」過程として見せる。誇張された魔法感（キラキラ・紫グロー）ではなく、静かな集中を演出する。

実装（CSSアニメーションのみ、self-contained）:
1. **音の波形フェーズ（0〜1.2s相当）**: 送信した10秒音声のダミー波形（15本のバー、高さをCSS変数でランダム初期化）が `--color-primary` の低彩度トーンで小さく上下する。`animation: waveform 1.1s ease-in-out infinite;` バー間に `animation-delay` を0.05s刻みでずらし、群れとして揺れて見せる。
2. **収束フェーズ（1.2s〜完了）**: 波形バーが中央に集約し、3.2節の「霧が晴れるインジケーター」の円3つへとモーフィング（`transform: scale()` と `opacity` のみをトランジション、`transition-property: transform, opacity` で明示指定。`transition: all` は使わない）。円は最初ぼやけた状態（blur 4px）から始まり、応答到着時に一気にblur 0へ収束する＝「像が結ばれる」瞬間を成功状態への遷移と同期させる。
3. 進行中テキストは1つだけ、typography原則に沿って `"むぎの様子を読み取っています…"` のように三点リーダ（`…`、`...`ではない）で表現。文言は3〜4秒ごとに1回だけ穏やかにクロスフェードで差し替え可能（最大2パターン）だが、点滅・pulseは使わない。
4. 完了までの目安が3秒を超える場合のみ、下部に極小のテキストで「もう少しで見立てが届きます」を追加表示。

`prefers-reduced-motion: reduce` 時: 波形アニメーションと収束モーフィングを無効化し、静的な3円（すべてmidトーン、blur無し）+ 進行中テキストのみを表示。

### 4.2 empty（未入力）

- 初回起動時、フィールドノートの「白紙ページ」を想起させるレイアウト。カード枠線は破線 (`border: 1px dashed var(--color-border)`) で「まだ何も書かれていない」ことを示す。
- 中央に犬の輪郭アイコン（線画）+ 「鳴き声・写真・状況を教えてください」という1行と、入力を促すテキストCTA。
- プレースホルダーは一般論ではなく、その子のプロフィールが未登録の場合は先にプロフィール登録（名前・犬種・年齢）への導線を優先表示する（7章「その子の名前」原則と連動）。

### 4.3 error（スキーマ失敗等）

- 赤系（`--color-urgent`）を使うが、緊急性フラグと混同しないよう**アイコンと文言で明確に区別**する（緊急性フラグ＝犬の安全、error＝システムの不具合であることをコピーで明示）。
- 文言は「エラーが発生しました」で終わらせず、原因の推定と次の行動を必ず添える（web-design-guidelines: エラーは修正手段/次の一手を含む）。例: 「見立ての生成に失敗しました。もう一度お試しいただくか、音声を録り直してみてください。」
- 再試行ボタンを主CTA位置に配置。フォーカスをエラーメッセージ先頭に移動する（フォームエラー時のフォーカス制御と同じ原則）。
- 部分的にデータが取れている場合（例: 画像は解析できたが音声が失敗）は、取得できた情報だけを低確信度カードとして出し、失敗箇所のみをエラー注記にする「部分的成功」表示を優先する（全滅表示にしない）。

### 4.4 success（見立てカード表示）

- 3章のコンポーネント構造をそのまま描画。
- カード出現時のみ、1回限りのステガーアニメーション（4.1節の収束の続きとして、CardHeader→PrimaryHypothesis→以下セクションの順に `animation-delay` を60ms刻みで設定、合計400ms以内に収める）。スクロール連動のフェードインは使わない（このカードは常に即時全体表示が前提のため）。

## 5. マイクロインタラクション / モーション

すべてのアニメーションは「なぜ動くのか」を1行で説明できるもののみ許可する。

| 要素 | モーション | 理由 | 実装 |
|---|---|---|---|
| loading→success遷移 | 波形→円の収束モーフィング | 処理が完了し理解が像を結んだことを1点に集約して伝える | `transform`/`opacity`のみ、220ms ease-out |
| カード初出現 | セクション単位のstagger | 情報の読む順序（仮説→確信度→観察→次の一手）を視覚的に誘導 | `animation-delay` 60ms刻み、合計400ms以内 |
| 観察チェックリストのタップ | チェックマークの軽い伸縮 (`scale(0.9)→1`) | ユーザー操作の即時フィードバック | 120ms ease-out、`transform`のみ |
| 次の一手CTAのホバー/フォーカス | 背景色を`--color-accent-soft`へ、影をわずかに強める | 操作可能であることの確認 | 150ms、`background-color`と`box-shadow`のみ |
| 緊急性フラグ帯の初出現 | 上からslide-inではなく、即時表示＋境界の1回だけの淡いハイライト | 緊急情報は遅延なく視認させる必要があり、演出で遅らせない | アニメーションほぼ無し（100ms fade程度） |
| 履歴差分バッジ | なし（静的） | 装飾的アニメーション不要 | — |

禁止: `animate-pulse`、`animate-bounce`の使用、`transition: all`の全要素適用、スクロールトリガーの装飾フェードイン全般。

`prefers-reduced-motion: reduce` 時は上表の全モーションを次に置き換える:
- loading→success: クロスフェードのみ（150ms opacity）
- stagger: 一括表示（delay 0）
- CTAホバー: 色変化のみ即時反映、transformは付けない

## 6. 表示モード別の振る舞い（inline / pip / fullscreen）

`window.openai.displayMode` と `requestDisplayMode` に応じて情報密度を切り替える。inlineが主戦場のため、**inlineで完結する体験**を最優先に設計し、fullscreenは「深掘り」の位置づけにする。

| 要素 | inline（既定） | pip | fullscreen |
|---|---|---|---|
| CardHeader | 名前・サムネ・時刻 | 同左（縮小） | 名前・サムネ・時刻＋犬種・月齢 |
| 主仮説 | 表示 | 表示 | 表示（拡大） |
| 次点仮説 | 最大2件のチップ | 非表示（主仮説のみ） | 全件表示 |
| 確信度インジケーター | 簡略版（円1つ、色のみ） | 簡略版 | フル版（3円＋限界文全文） |
| 観察チェックリスト | 上位2項目のみ＋「もっと見る」 | 非表示 | 全項目＋各項目の解説文 |
| 次の一手 | 主CTA1つのみ | 主CTA1つのみ | 主CTA＋サブCTA全て |
| 追加質問 | 1問のみ、テキスト誘導 | 非表示 | チャット風の複数往復表示可 |
| 履歴差分 | 非表示（fullscreenへの導線のみ） | 非表示 | グラフ／タイムライン表示 |
| 「fullscreenで見る」導線 | カード右上に小さく常設 | 非表示 | — |

- inline→fullscreenの遷移は `requestDisplayMode({ mode: "fullscreen" })` をユーザー操作（「もっと見る」タップ）でのみ呼び出す。自動昇格はしない。
- pipは「会話を進めながら見立てを横目に確認する」用途に限定し、情報を主仮説＋CTAのみに絞る（読む負荷を最小化）。
- fullscreenでのみ履歴のタイムライン（過去の見立てカードのサムネイル列）を表示し、「その子の記録が積み重なっている」実感を作る。

## 7. 親しみを生む工夫

### 7.1 名前と写真の扱い

- 本文中で犬を指す際は必ず個体名を使う（「あなたの犬」「対象犬」等の一般化表現を禁止）。名前未登録時はempty状態で先に名前入力を促す。
- 写真サムネイルは正方形トリミングで`--radius-card`と同じ16pxの角丸（顔認識トリミングは行わず、ユーザーが選んだ構図をそのまま尊重する）。
- 履歴が3件以上たまったら、fullscreenのCardHeaderに「〇〇との記録：12回目」のような蓄積を示す一言を出し、使い続けるほど手帳が育っていく感覚を作る。

### 7.2 トーン&マナー

- 敬体（です・ます）を基本としつつ、体言止めや事務的な箇条書きだけの文にしない。1文は主語（その子の名前 or 状況）を明示した完結文にする。
- 語りかけは「一緒に見ていきましょう」トーン。上から診断する語調（「〜と判定されました」）を避ける。
- 絵文字は装飾として使わない（3.3節の通りアイコンは線画SVG）。ただし犬種名の横などプロフィール情報には控えめな肉球アイコン1つ程度は許容。

### 7.3 コピーライティング原則

| 原則 | 良い例 | 避ける例 |
|---|---|---|
| 断定しない語尾で統一 | 「警戒している可能性があります」 | 「警戒しています」 |
| 犬のセリフを作らない | 「体を低くして視線を外している様子です」 | 「こわいよ、と言っています」 |
| 理由を伴う限界の明示 | 「音声のみのため、体の硬さまでは読み取れていません」 | 「確信度：低」だけを表示 |
| 次の行動を必ず添える | 「刺激を与えず、少し距離を取ってみてください」 | 「様子を見てください」（具体性なし） |
| 二人称で語りかける | 「〇〇の耳の向きを見てみましょう」 | 「耳の向きを確認する必要がある」 |

## 9. やってはいけないこと（アンチパターン）

### コンテンツ・トーン
- 犬のセリフ化・断定翻訳（「お腹すいたよ、と言っています」等）。PawLensの中核思想への違反であり最優先禁止。
- 確信度・限界文を省略した仮説単独表示。
- 「診断」「判定」「〜病の可能性」等、医療/行動診断を想起させる語彙。
- 緊急性フラグの過剰演出（点滅・警告音・全画面赤塗り）による不安の煽り立て。安全に導線を張ることと恐怖訴求は別物。

### ビジュアル
- 白背景＋紫〜青のグラデーション、紫をAI機能の記号として使うこと。
- Inter / Roboto / Arial / システムフォント単体のみのスタック。フォントファミリー間のウェイト差が200以下の組み合わせ。
- 均等3列カードグリッドの反復（状態仮説・観察ポイント・次の一手を同一グリッドで並べない）。
- 確信度を単純な進捗バー（0-100%塗りつぶし）で表現すること＝断定的に見えるため禁止。3.2節の独自インジケーターを使う。
- 全カード同一の`shadow-lg`＋`rounded-2xl`の機械的適用。
- 装飾目的だけの半透明カード・過剰なぼかし背景。

### モーション
- `animate-pulse` / `animate-bounce` の多用。
- loading演出以外での常時アニメーション（カード内要素が常にゆらゆら動く等）。
- スクロールトリガーの全要素フェードイン（本ウィジェットはスクロール前提のLPではないため無意味）。
- `transition: all` の全要素一律適用。

### 構造
- 表示モードに関わらず全情報を常に全表示すること（inlineが情報過多で読みにくくなる）。
- fullscreenへの自動昇格・無断遷移。
- 4状態（loading/empty/error/success）のいずれかを実装省略すること。特にerrorの「部分的成功」表示の省略（失敗時に全滅表示にしてしまう）。

## 10. シグネチャー・インタラクション「断定回避リビール」（デモの顔）

> **目的**: PawLensの中核思想「犬語を断定翻訳しない＝不確実性の通訳」を、説明ではなく**1つのモーションで体感**させる。3分デモで最もスクリーンショットされ、審査員の記憶に残る"アハ"の瞬間として設計する。アイデア品質の突出を「地味さ」で埋もれさせないための、意図的なドラマ。

> **⚠️ 2026-07-16 デスコープ決定（red-team M-3）**: プロトタイプにはこの演出が**未実装**であることが実測で確認された。MVP提出版の正式仕様は **§10.2 で定義済みの静的 before→after 表現**（「よくある翻訳: ~~こわいよ！~~ → PawLensの見立て」の一行対比）とし、§10.1 のフルアニメーション版は**提出後の拡張として実装しない**。デモ動画・台本（positioning-uplift §6）も静的版を前提に組む。思想の伝達力は保持され、実装コストは約1/10になる。

### 10.1 体験のストーリー（3.5秒）※提出後の拡張（MVPでは実装しない）
loading の収束フェーズ（design仕様4.1）に続けて、success への遷移の**直前0.8秒だけ**差し込む。

```
波形が集束する
  ↓
【0.0s】中央に “ゴースト吹き出し” が淡く浮かびかける
        —— よくある犬語翻訳が言いそうな一言（例: 「こわいよ！」）が
           ぼやけた薄いグレーで、半分だけ現れる
  ↓
【0.4s】その吹き出しに、すっと横線が引かれる（消すのではなく "選ばない" 所作）
        同時に小さなラベル: 「決めつけません」
  ↓
【0.6s】ゴーストが霧のように上へ散り、
        その粒子が「霧が晴れるインジケーター」の3円へと吸い込まれる
  ↓
【0.8s】success カードが結像：
        主仮説「"警戒" の可能性・確信度 中」＋観察ポイント＋次の一手
```

**言い切れる一言を、あえて言い切らない。** この"引き算の所作"を可視化することが、競合の「犬語翻訳」との差を一撃で伝える。

### 10.2 厳守する安全設計（最重要）
ゴースト吹き出しは「PawLensが言っていること」に**絶対に見えてはならない**。誤解・切り抜き拡散を防ぐため:

- ゴーストは常に **`--color-text-faint` の薄いグレー＋ blur 2px** で、本文より明確に弱い視覚重み。カラーの吹き出し・犬アイコン付き吹き出しにしない。
- 出現と同時に**必ず取り消し線と「決めつけません／We don't put words in their mouth」ラベルを伴う**。ラベル無しでゴーストだけが見える単独フレームを1フレームも作らない。
- ゴーストの文言は**固定の1パターン**（例: 「こわいよ！」/ EN: "I'm scared!"）にとどめ、実際の入力から犬のセリフを生成しない（＝セリフ生成ロジックは実装しない。これは静的な"反例テンプレート"）。
- 滞留時間は0.8秒以内。長く見せて弄ばない。
- `prefers-reduced-motion: reduce` 時: モーションを廃し、**「よくある翻訳: ~~こわいよ！~~ → PawLensの見立て」という静的な before→after の一行**に置き換える（意味は保持、演出のみ除去）。
- アクセシビリティ: ゴーストは `aria-hidden="true"`（スクリーンリーダーには読ませない）。success結像後の本文のみ `aria-live` で通知。

### 10.3 実装メモ（self-contained）
- ゴースト吹き出し・取り消し線・散逸は `opacity` / `transform` / `clip-path` のみで表現（`transition: all` 禁止）。
- 散逸パーティクルは擬似要素＋少数のspanで表現し、GIF/外部アセットを使わない。
- この演出は `analyze_dog_signal` の応答到着に同期して1回だけ発火。再解析時も毎回出す（デモで確実に見せられるように）。ただしユーザー設定 or `widgetState` で2回目以降スキップ可能にしてもよい（好みで）。

### 10.4 デモ台本での使い方（3分動画の 0:20–0:40 "outcome & differentiation" ビート）
- ナレーション例（EN）: *"A dog translator would just say 'I'm scared.' PawLens refuses to — and shows you what to actually look at."*
- **静的リビール（before→after の一行対比）を一時停止で強調**し、「断定しないという選択」をデモの主役にする。
- 画面には英語字幕でラベル（"We don't put words in their mouth."）を必ず出す（グローバル審査対応）。
